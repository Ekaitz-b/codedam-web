Bases de Datos

SQL, MySQL y Diseño de Bases de Datos

Índice de Contenidos

    Introducción a Bases de Datos

    Una base de datos es un conjunto organizado de información estructurada que se almacena electrónicamente en un sistema informático. Las bases de datos relacionales organizan los datos en tablas con filas y columnas.

    Conceptos Fundamentales

    • Tabla: Conjunto de datos organizados en filas y columnas
    • Fila (Registro): Una entrada individual en una tabla
    • Columna (Campo): Un atributo de los datos
    • Clave Primaria (Primary Key): Identifica únicamente cada registro
    • Clave Foránea (Foreign Key): Referencia a la clave primaria de otra tabla

    Sistemas Gestores de Bases de Datos (SGBD)

    Los más utilizados en DAM son:

    • MySQL: Popular, código abierto, ideal para aplicaciones web
    • PostgreSQL: Robusto, con características avanzadas
    • Oracle: Potente, usado en entornos empresariales
    • SQL Server: De Microsoft, integrado con .NET

    SQL Básico

    SQL (Structured Query Language) es el lenguaje estándar para gestionar bases de datos relacionales.

    CREATE - Crear Tablas

    CREATE DATABASE tienda_online; USE tienda_online; CREATE TABLE clientes ( id_cliente INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100) NOT NULL, email VARCHAR(100) UNIQUE, telefono VARCHAR(15), fecha_registro DATE DEFAULT CURRENT_DATE ); CREATE TABLE productos ( id_producto INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(200) NOT NULL, precio DECIMAL(10, 2) NOT NULL, stock INT DEFAULT 0, categoria VARCHAR(50) ); CREATE TABLE pedidos ( id_pedido INT PRIMARY KEY AUTO_INCREMENT, id_cliente INT, fecha_pedido DATETIME DEFAULT CURRENT_TIMESTAMP, total DECIMAL(10, 2), FOREIGN KEY (id_cliente) REFERENCES clientes(id_cliente) );

    INSERT - Insertar Datos

    -- Insertar un registro INSERT INTO clientes (nombre, email, telefono) VALUES ('Ana García', 'ana@email.com', '600123456'); -- Insertar múltiples registros INSERT INTO productos (nombre, precio, stock, categoria) VALUES ('Portátil HP', 599.99, 15, 'Informática'), ('Mouse Logitech', 25.50, 50, 'Accesorios'), ('Teclado Mecánico', 89.99, 30, 'Accesorios');

    SELECT - Consultar Datos

    -- Seleccionar todos los campos SELECT * FROM clientes; -- Seleccionar campos específicos SELECT nombre, email FROM clientes; -- Con condiciones (WHERE) SELECT * FROM productos WHERE precio > 50 AND stock > 0; -- Ordenar resultados (ORDER BY) SELECT * FROM productos ORDER BY precio DESC; -- Limitar resultados SELECT * FROM productos LIMIT 10; -- Búsqueda con patrón (LIKE) SELECT * FROM clientes WHERE nombre LIKE 'Ana%'; -- Valores únicos (DISTINCT) SELECT DISTINCT categoria FROM productos;

    UPDATE - Actualizar Datos

    -- Actualizar un registro UPDATE productos SET precio = 549.99, stock = 20 WHERE id_producto = 1; -- Actualizar múltiples registros UPDATE productos SET categoria = 'Hardware' WHERE categoria = 'Informática'; -- Incrementar valores UPDATE productos SET precio = precio * 1.10 WHERE categoria = 'Accesorios';

    DELETE - Eliminar Datos

    -- Eliminar registros específicos DELETE FROM productos WHERE stock = 0; -- CUIDADO: Esto elimina TODOS los registros DELETE FROM productos; -- Mejor usar TRUNCATE para vaciar tabla TRUNCATE TABLE productos;

    SQL Avanzado

    JOINs - Unir Tablas

    Los JOINs permiten combinar datos de múltiples tablas relacionadas.

    -- INNER JOIN: Solo registros que coinciden en ambas tablas SELECT clientes.nombre, pedidos.fecha_pedido, pedidos.total FROM pedidos INNER JOIN clientes ON pedidos.id_cliente = clientes.id_cliente; -- LEFT JOIN: Todos los de la izquierda + coincidentes de la derecha SELECT clientes.nombre, pedidos.total FROM clientes LEFT JOIN pedidos ON clientes.id_cliente = pedidos.id_cliente; -- RIGHT JOIN: Todos los de la derecha + coincidentes de la izquierda SELECT productos.nombre, detalle_pedido.cantidad FROM productos RIGHT JOIN detalle_pedido ON productos.id_producto = detalle_pedido.id_producto; -- JOIN de múltiples tablas SELECT clientes.nombre AS cliente, productos.nombre AS producto, detalle_pedido.cantidad, pedidos.fecha_pedido FROM pedidos INNER JOIN clientes ON pedidos.id_cliente = clientes.id_cliente INNER JOIN detalle_pedido ON pedidos.id_pedido = detalle_pedido.id_pedido INNER JOIN productos ON detalle_pedido.id_producto = productos.id_producto;

    Funciones de Agregación

    -- COUNT: Contar registros SELECT COUNT(*) AS total_clientes FROM clientes; -- SUM: Sumar valores SELECT SUM(total) AS ventas_totales FROM pedidos; -- AVG: Promedio SELECT AVG(precio) AS precio_medio FROM productos; -- MAX y MIN SELECT MAX(precio) AS mas_caro, MIN(precio) AS mas_barato FROM productos; -- GROUP BY: Agrupar resultados SELECT categoria, COUNT(*) AS cantidad, AVG(precio) AS precio_medio FROM productos GROUP BY categoria; -- HAVING: Filtrar grupos (como WHERE pero para GROUP BY) SELECT categoria, AVG(precio) AS precio_medio FROM productos GROUP BY categoria HAVING AVG(precio) > 50;

    Subconsultas

    -- Subconsulta en WHERE SELECT nombre, precio FROM productos WHERE precio > (SELECT AVG(precio) FROM productos); -- Subconsulta con IN SELECT nombre FROM clientes WHERE id_cliente IN ( SELECT id_cliente FROM pedidos WHERE total > 100 ); -- Subconsulta en FROM SELECT categoria, AVG(precio) AS promedio FROM ( SELECT * FROM productos WHERE stock > 0 ) AS productos_disponibles GROUP BY categoria;

    Diseño de Bases de Datos

    Modelo Entidad-Relación (E-R)

    El modelo E-R es una herramienta para diseñar bases de datos de forma visual antes de implementarlas.

    • Entidades: Objetos del mundo real (Cliente, Producto, Pedido)
    • Atributos: Características de las entidades (nombre, precio, fecha)
    • Relaciones: Vínculos entre entidades

    Tipos de Relaciones

    • 1:1 (Uno a Uno): Un registro de A se relaciona con un registro de B
    • 1:N (Uno a Muchos): Un registro de A se relaciona con varios de B
    • N:M (Muchos a Muchos): Varios de A se relacionan con varios de B (requiere tabla intermedia)

    Ejemplo 1:N: Un cliente puede tener muchos pedidos, pero cada pedido pertenece a un solo cliente.

    Normalización

    Proceso de organizar datos para reducir redundancia y mejorar la integridad.

    Primera Forma Normal (1FN)

    • Cada columna debe contener valores atómicos (indivisibles)
    • No debe haber grupos repetitivos
    • Debe existir una clave primaria

    Segunda Forma Normal (2FN)

    • Debe cumplir 1FN
    • Todos los atributos no clave deben depender completamente de la clave primaria

    Tercera Forma Normal (3FN)

    • Debe cumplir 2FN
    • No debe haber dependencias transitivas (atributos que dependan de otros atributos no clave)
    -- Ejemplo de normalización -- SIN NORMALIZAR (1 tabla con redundancia) CREATE TABLE pedidos_malo ( id_pedido INT PRIMARY KEY, cliente_nombre VARCHAR(100), cliente_email VARCHAR(100), producto_nombre VARCHAR(100), producto_precio DECIMAL(10,2) ); -- NORMALIZADO (3 tablas relacionadas) CREATE TABLE clientes ( id_cliente INT PRIMARY KEY, nombre VARCHAR(100), email VARCHAR(100) ); CREATE TABLE productos ( id_producto INT PRIMARY KEY, nombre VARCHAR(100), precio DECIMAL(10,2) ); CREATE TABLE pedidos ( id_pedido INT PRIMARY KEY, id_cliente INT, id_producto INT, FOREIGN KEY (id_cliente) REFERENCES clientes(id_cliente), FOREIGN KEY (id_producto) REFERENCES productos(id_producto) );

    Optimización y Buenas Prácticas

    Índices

    Los índices mejoran la velocidad de las consultas, especialmente en tablas grandes.

    -- Crear índice CREATE INDEX idx_email ON clientes(email); -- Índice compuesto CREATE INDEX idx_nombre_categoria ON productos(nombre, categoria); -- Ver índices de una tabla SHOW INDEX FROM clientes;

    Transacciones

    Garantizan que un conjunto de operaciones se ejecuten completamente o no se ejecuten.

    START TRANSACTION; UPDATE cuentas SET saldo = saldo - 100 WHERE id_cuenta = 1; UPDATE cuentas SET saldo = saldo + 100 WHERE id_cuenta = 2; -- Si todo está bien: COMMIT; -- Si hay error: ROLLBACK;

    Consejos y Mejores Prácticas

    • Usa nombres descriptivos para tablas y columnas
    • Siempre define claves primarias
    • Evita NULL cuando sea posible (usa DEFAULT)
    • Usa tipos de datos apropiados (no VARCHAR(255) para todo)
    • Normaliza pero no en exceso (denormalización controlada)
    • Crea índices en columnas de búsqueda frecuente
    • Usa transacciones para operaciones críticas
    • Haz backups regulares de tus bases de datos

    Error común: No usar WHERE en UPDATE o DELETE puede actualizar/eliminar TODOS los registros. Siempre verifica tus condiciones.
