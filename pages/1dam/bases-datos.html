    Cardinalidad de Relaciones

    Define cuГЎntas instancias de una entidad se pueden relacionar con instancias de otra.

    1:1 (Uno a Uno)

    Un registro de A se relaciona con exactamente un registro de B, y viceversa.

    Ejemplo: PERSONA (1) ---tiene---> (1) PASAPORTE - Una persona tiene un Гәnico pasaporte - Un pasaporte pertenece a una Гәnica persona CREATE TABLE personas ( id_persona INT PRIMARY KEY, nombre VARCHAR(100), dni VARCHAR(9) UNIQUE ); CREATE TABLE pasaportes ( id_pasaporte INT PRIMARY KEY, numero_pasaporte VARCHAR(20), fecha_expedicion DATE, id_persona INT UNIQUE, -- UNIQUE asegura la relaciГіn 1:1 FOREIGN KEY (id_persona) REFERENCES personas(id_persona) );

    1:N (Uno a Muchos) - MГҒS COMГҡN

    Un registro de A se puede relacionar con varios registros de B, pero un registro de B solo se relaciona con uno de A.

    Ejemplo: CLIENTE (1) ---realiza---> (N) PEDIDOS - Un cliente puede hacer muchos pedidos - Cada pedido pertenece a un solo cliente CREATE TABLE clientes ( id_cliente INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100), email VARCHAR(100) ); CREATE TABLE pedidos ( id_pedido INT PRIMARY KEY AUTO_INCREMENT, fecha DATE, total DECIMAL(10,2), id_cliente INT, -- La FK va en el lado "muchos" FOREIGN KEY (id_cliente) REFERENCES clientes(id_cliente) ); -- Otros ejemplos 1:N: -- DEPARTAMENTO (1) --- (N) EMPLEADOS -- CATEGORГҚA (1) --- (N) PRODUCTOS -- AUTOR (1) --- (N) LIBROS

    N:M (Muchos a Muchos)

    Varios registros de A se relacionan con varios de B, y viceversa. Requiere una tabla intermedia.

    Ejemplo: ESTUDIANTE (N) ---cursa---> (M) ASIGNATURA - Un estudiante puede cursar muchas asignaturas - Una asignatura puede tener muchos estudiantes -- SOLUCIГ“N: Tabla intermedia "matriculas" CREATE TABLE estudiantes ( id_estudiante INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100), email VARCHAR(100) ); CREATE TABLE asignaturas ( id_asignatura INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100), creditos INT ); -- Tabla intermedia (convierte N:M en dos relaciones 1:N) CREATE TABLE matriculas ( id_matricula INT PRIMARY KEY AUTO_INCREMENT, id_estudiante INT, id_asignatura INT, fecha_matricula DATE, nota DECIMAL(4,2), FOREIGN KEY (id_estudiante) REFERENCES estudiantes(id_estudiante), FOREIGN KEY (id_asignatura) REFERENCES asignaturas(id_asignatura), UNIQUE(id_estudiante, id_asignatura) -- Un estudiante no puede matricularse 2 veces en la misma asignatura ); -- Otros ejemplos N:M: -- PRODUCTO (N) --- (M) PEDIDO вҶ’ tabla "detalle_pedido" -- ACTOR (N) --- (M) PELГҚCULA вҶ’ tabla "reparto" -- MГүDICO (N) --- (M) PACIENTE вҶ’ tabla "citas"

    Subconsultas (Subqueries)

    Una subconsulta es una query SELECT dentro de otra query. Puede aparecer en clГЎusulas SELECT, WHERE, FROM, HAVING.

    Ejemplo 8: Subconsulta en WHERE 
    -- Productos con precio MAYOR al precio promedio
SELECT
    nombre,
    precio
FROM productos
WHERE precio > (SELECT AVG(precio) FROM productos);  -- Subconsulta devuelve un valor

-- Desglose:
-- 1. Subconsulta calcula: AVG(precio) = 271.50
-- 2. Query externa busca productos con precio > 271.50

-- Clientes que han gastado MГҒS que el promedio
SELECT
    c.nombre,
    SUM(p.total) AS total_gastado
FROM clientes c
INNER JOIN pedidos p ON c.id_cliente = p.id_cliente
GROUP BY c.id_cliente, c.nombre
HAVING SUM(p.total) > (
    SELECT AVG(total_por_cliente)
    FROM (
        SELECT SUM(total) AS total_por_cliente
        FROM pedidos
        GROUP BY id_cliente
    ) AS subquery
);

    • Olvidar la condiciГіn ON en JOIN: Sin ON obtienes un producto cartesiano (todas las combinaciones)
    • Usar WHERE en vez de HAVING: WHERE filtra filas individuales, HAVING filtra grupos
    • No incluir columnas en GROUP BY: Todas las columnas no agregadas deben estar en GROUP BY
    • Subconsultas sin alias: Tablas derivadas en FROM DEBEN tener alias
    • Comparar con NULL usando =: Usa IS NULL o IS NOT NULL, nunca = NULL
    • LEFT JOIN con WHERE que elimina NULLs: WHERE tabla_derecha.columna = valor convierte LEFT JOIN en INNER JOIN
    • Subconsultas que devuelven mГәltiples filas con =: Usa IN en vez de = si subconsulta puede devolver varios valores
    рҹ’Ў Pro Tips - SQL Intermedio

    • Prefiere JOIN sobre subconsultas cuando sea posible (suele ser mГЎs rГЎpido)
    • Alias descriptivos mejoran legibilidad: c para clientes, p para pedidos
    • LEFT JOIN + WHERE IS NULL es el patrГіn para "elementos sin relaciГіn"
    • Usa IFNULL() o COALESCE() para manejar NULLs en cГЎlculos
    • GROUP_CONCAT() es Гәtil para concatenar valores de un grupo
    • CASE WHEN permite lГіgica condicional dentro de SELECT
    • Siempre usa LIMIT cuando testeas queries nuevas en producciГіn
    • Formatea queries complejas: una clГЎusula por lГӯnea, indenta subconsultas
    • EXISTS puede ser mГЎs rГЎpido que IN con subconsultas grandes

    NormalizaciГіn de Bases de Datos

    ВҝQuГ© es la NormalizaciГіn?

    La normalizaciГіn es el proceso de organizar datos en una base de datos relacional para:

    • Eliminar redundancia: No duplicar informaciГіn innecesariamente
    • Evitar anomalГӯas: Prevenir problemas al insertar, actualizar o eliminar datos
    • Mantener integridad: Asegurar que los datos sean consistentes
    • Facilitar mantenimiento: Hacer cambios en el esquema mГЎs sencillos

    Existen varias formas normales. Las mГЎs importantes son: 1FN, 2FN y 3FN.

    Primera Forma Normal (1FN)

    Una tabla estГЎ en 1FN si:

    • Todos los valores son atГіmicos (indivisibles, un solo valor por celda)
    • No hay grupos repetidos de columnas
    • Cada fila es Гәnica (tiene clave primaria)
    вқҢ Tabla que NO cumple 1FN 
    -- Problema: Columna "telefonos" contiene MГҡLTIPLES valores (no atГіmica)
-- Problema: Columna "productos_comprados" es una lista

    Segunda Forma Normal (2FN)

    Una tabla estГЎ en 2FN si:

    • EstГЎ en 1FN
    • Todos los atributos no clave dependen completamente de la clave primaria (no de parte de ella)
    • Solo aplica a tablas con clave primaria compuesta (mГәltiples columnas)
    вқҢ Tabla que NO cumple 2FN 
    -- Tabla con clave primaria compuesta (id_pedido + id_producto)
CREATE TABLE detalle_pedidos_mal (
    id_pedido INT,
    id_producto INT,
    cantidad INT,
    precio_unitario DECIMAL(10,2),
    nombre_producto VARCHAR(100),  -- вқҢ Depende SOLO de id_producto (no de la clave completa)
    categoria VARCHAR(50),         -- вқҢ Depende SOLO de id_producto
    nombre_cliente VARCHAR(100),   -- вқҢ Depende SOLO de id_pedido
    PRIMARY KEY (id_pedido, id_producto)
);

-- Ejemplo de datos:
-- +----------+------------+----------+------------------+-----------------+-----------+----------------+
-- | id_pedido| id_producto| cantidad | precio_unitario  | nombre_producto | categoria | nombre_cliente |

    вң… Tabla en 2FN - SoluciГіn Paso a Paso

    Tercera Forma Normal (3FN)

    Una tabla estГЎ en 3FN si:

    • EstГЎ en 2FN
    • No tiene dependencias transitivas: atributos no clave NO dependen de otros atributos no clave

    вҡ пёҸ Errores Comunes en NormalizaciГіn
    • Sobre-normalizaciГіn: Dividir DEMASIADO puede hacer queries muy complejas. Encuentra el balance.
    • Olvidar FKs: Al separar tablas, SIEMPRE crea Foreign Keys para mantener integridad
    • No identificar dependencias transitivas: Revisa si columna A вҶ’ columna B вҶ’ columna C
    • Confundir 2FN y 3FN: 2FN es para claves compuestas, 3FN es para dependencias entre no-claves
    • Valores separados por comas: "Item1, Item2" en una columna SIEMPRE viola 1FN
    • Duplicar datos en vez de usar relaciones: Si repites info, probablemente necesitas otra tabla
    рҹ’Ў Pro Tips - NormalizaciГіn
    • Regla de oro: Si repites datos, normaliza. Cada dato debe estar en UN solo lugar.
    • La normalizaciГіn reduce espacio pero puede aumentar complejidad de queries (mГЎs JOINs)
    • Para aplicaciones de lectura intensiva, a veces se desnormaliza intencionalmente (cachГ©, reporting)
    • Usa diagramas E-R antes de implementar para visualizar relaciones
    • Las tablas intermedias (N:M) suelen tener clave compuesta вҶ’ verifica 2FN
    • 3FN es suficiente para la mayorГӯa de aplicaciones. BCNF y 4FN son raramente necesarias.
    • PregГәntate: "ВҝSi actualizo dato X, debo tocar mГәltiples filas?" вҶ’ Si sГӯ, normaliza mГЎs
    • Normaliza en diseГұo; considera vistas o procedures para queries complejas frecuentes

    ВҝQuГ© son los ГҚndices?

    Un Гӯndice es una estructura de datos que mejora la velocidad de las consultas en una tabla. Funciona como el Гӯndice de un libro: en vez de leer todo el libro para encontrar un tema, vas directamente a la pГЎgina indicada.

    • Ventaja: Consultas SELECT mucho mГЎs rГЎpidas (especialmente con WHERE, JOIN, ORDER BY)
    • Desventaja: Ralentiza INSERT, UPDATE, DELETE (el Гӯndice debe actualizarse)
    • Espacio: Los Гӯndices ocupan espacio adicional en disco

    MySQL crea automГЎticamente Гӯndices para PRIMARY KEY y UNIQUE. TГә puedes crear Гӯndices adicionales en columnas que usas frecuentemente en bГәsquedas.

    Transacciones

    ВҝQuГ© es una TransacciГіn?

    Una transacciГіn es un conjunto de operaciones SQL que se ejecutan como una unidad atГіmica: o se ejecutan TODAS con Г©xito, o NO se ejecuta NINGUNA.

    Ejemplo del mundo real: Transferencia bancaria entre dos cuentas:

    • Paso 1: Restar dinero de cuenta A
    • Paso 2: Sumar dinero a cuenta B

    Si el paso 1 funciona pero el paso 2 falla (error de red, caГӯda del servidor), el dinero desaparece. Transacciones garantizan que AMBOS pasos ocurran o NINGUNO.

    Propiedades ACID

    Las transacciones deben cumplir 4 propiedades conocidas como ACID:

    ACID - DefiniciГіn 
    -- A - ATOMICITY (Atomicidad)
-- La transacciГіn es indivisible: TODO o NADA
-- Si una operaciГіn falla, se revierten TODAS las anteriores

-- C - CONSISTENCY (Consistencia)
-- La BD pasa de un estado vГЎlido a otro estado vГЎlido
-- No se violan constraints (PRIMARY KEY, FOREIGN KEY, CHECK, etc.)

-- I - ISOLATION (Aislamiento)
-- Transacciones concurrentes no interfieren entre sГӯ
-- Cada transacciГіn actГәa como si fuera la Гәnica ejecutГЎndose

-- D - DURABILITY (Durabilidad)
-- Una vez hecho COMMIT, los cambios son PERMANENTES
-- Sobreviven a fallos del sistema (crash, reinicio)

    Niveles de Aislamiento (Isolation Levels)

    Los niveles de aislamiento controlan quГ© tan "aisladas" estГЎn las transacciones concurrentes. Mayor aislamiento = mГЎs seguridad, pero menos concurrencia (mГЎs lento).

    Niveles de Aislamiento en MySQL 
    -- 4 niveles estГЎndar (de menos a mГЎs aislamiento):

-- 1. READ UNCOMMITTED (lectura sucia)
-- Puede leer cambios NO confirmados de otras transacciones
-- Problema: "Dirty read" (leer datos que pueden revertirse)

-- 2. READ COMMITTED (lectura confirmada) - MГЎs comГәn en producciГіn
-- Solo lee cambios confirmados (despuГ©s de COMMIT)
-- Problema: "Non-repeatable read" (misma query da resultados diferentes en la misma transacciГіn)
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 3. REPEATABLE READ (lectura repetible) - DEFAULT en MySQL
-- Misma query en la transacciГіn da siempre el mismo resultado
-- Problema: "Phantom read" (nuevas filas insertadas por otras transacciones pueden aparecer)
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 4. SERIALIZABLE (serializable) - MГЎximo aislamiento
-- Transacciones se ejecutan completamente aisladas (como si fueran secuenciales)
-- Sin problemas de concurrencia, pero MUY LENTO

-- Ver nivel actual

-- Cambiar nivel globalmente (para todas las conexiones nuevas)

    • DDL dentro de transacciones: CREATE TABLE, ALTER TABLE hacen COMMIT implГӯcito (terminan la transacciГіn).
    • Asumir que AUTOCOMMIT estГЎ off: En MySQL AUTOCOMMIT=1 por defecto. Usa START TRANSACTION explГӯcitamente.
    • Leer datos sin transacciГіn: En aplicaciones concurrentes, dos SELECTs consecutivos pueden dar resultados diferentes.

    • En aplicaciones web, NUNCA dejes transacciones abiertas entre requests HTTP.
    • Para operaciones masivas (ej: importar 1M de filas), considera hacer COMMIT cada N filas (ej: cada 1000).
    • FOR UPDATE: Bloquea filas en SELECT para prevenir modificaciones concurrentes: SELECT * FROM cuentas WHERE id = 1 FOR UPDATE;
    • Usa procedimientos almacenados para lГіgica compleja de transacciones (menos latencia red).

    • MySQL detecta y resuelve deadlocks automГЎticamente (revierte una transacciГіn). Tu app debe reintentar.

    • En producciГіn, monitorea transacciones largas con SHOW PROCESSLIST;

    Ejercicio 13 (Bonus): Trigger y Procedimiento Almacenado

    Enunciado: Crea un procedimiento almacenado para registrar prГ©stamos y un trigger que actualice automГЎticamente la disponibilidad del libro.

    SoluciГіn 
    -- Crear procedimiento almacenado para registrar prГ©stamo
DELIMITER //

CREATE PROCEDURE registrar_prestamo(
    IN p_id_libro INT,
    IN p_id_socio INT,
    IN p_dias_plazo INT
)
BEGIN
    DECLARE libro_disponible BOOLEAN;

    -- Verificar disponibilidad
    SELECT disponible INTO libro_disponible
    FROM libros
    WHERE id_libro = p_id_libro;

    IF libro_disponible = FALSE THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Libro no disponible';
    END IF;

    -- Registrar prГ©stamo en transacciГіn
    START TRANSACTION;

    INSERT INTO prestamos (id_libro, id_socio, fecha_prestamo, fecha_devolucion_prevista)
    VALUES (p_id_libro, p_id_socio, CURDATE(), DATE_ADD(CURDATE(), INTERVAL p_dias_plazo DAY));

    UPDATE libros SET disponible = FALSE WHERE id_libro = p_id_libro;

    COMMIT;
END //

DELIMITER ;

-- Crear trigger para actualizar disponibilidad al devolver
DELIMITER //

CREATE TRIGGER after_devolucion
AFTER UPDATE ON prestamos
FOR EACH ROW
BEGIN
    -- Si se actualiza fecha_devolucion_real (estaba NULL, ahora tiene valor)
    IF OLD.fecha_devolucion_real IS NULL AND NEW.fecha_devolucion_real IS NOT NULL THEN
        UPDATE libros SET disponible = TRUE WHERE id_libro = NEW.id_libro;
    END IF;
END //

DELIMITER ;

-- Usar el procedimiento
