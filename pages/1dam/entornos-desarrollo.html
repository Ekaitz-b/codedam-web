Introducción al Desarrollo de Software

El módulo de Entornos de Desarrollo enseña las herramientas y metodologías necesarias para desarrollar software de forma profesional, incluyendo control de versiones, testing, debugging y trabajo en equipo.

Ciclo de Vida del Software (SDLC)

Fases del Desarrollo

  1. Análisis de Requisitos: Definir qué debe hacer el software
  2. Diseño: Arquitectura, diagramas, estructuras de datos
  3. Implementación: Escribir el código
  4. Pruebas: Testing unitario, integración, sistema
  5. Despliegue: Poner en producción
  6. Mantenimiento: Corrección de errores, mejoras

Modelos de Desarrollo

  • Cascada (Waterfall): Fases secuenciales, una tras otra. Poco flexible
  • Iterativo: Desarrollo en ciclos repetitivos
  • Incremental: Añadir funcionalidad gradualmente
  • Ágil: Iteraciones cortas, flexibilidad, colaboración continua

Modelo en Cascada vs Ágil

Cascada Ágil
Fases secuenciales Iteraciones cortas (sprints)
Requisitos fijos Requisitos pueden cambiar
Documentación extensa Software funcionando
Cliente ve al final Feedback continuo

Control de Versiones con Git

Git es el sistema de control de versiones más utilizado en el mundo. Permite rastrear cambios, colaborar en equipo y mantener un historial completo del proyecto.

Por qué usar Git

  • Historial completo de todos los cambios
  • Trabajo en paralelo con branches
  • Colaboración en equipo
  • Backup del código en repositorios remotos
  • Revertir errores fácilmente

Conceptos Fundamentales

  • Repository: Carpeta con historial de versiones
  • Commit: Instantánea del proyecto en un momento
  • Branch: Línea de desarrollo paralela
  • Merge: Fusionar ramas
  • Remote: Repositorio en servidor (GitHub, GitLab)
  • Clone: Copiar repositorio remoto a local
  • Pull: Descargar cambios del remoto
  • Push: Subir cambios al remoto

1. Configuración Inicial

# Configurar identidad (solo una vez) git config --global user.name "Tu Nombre" git config --global user.email "tu@email.com" # Ver configuración git config --list # Editor por defecto git config --global core.editor "code --wait"

2. Crear y Clonar Repositorios

# Crear repositorio nuevo mkdir mi-proyecto cd mi-proyecto git init # Clonar repositorio existente git clone https://github.com/usuario/repo.git # Clonar en carpeta específica git clone https://github.com/usuario/repo.git mi-carpeta

3. Comandos Básicos - Add, Commit, Status

# Ver estado del repositorio git status # Ver cambios en detalle git diff # Añadir archivo al staging area git add archivo.java # Añadir todos los archivos modificados git add . # Añadir solo archivos .java git add *.java # Hacer commit git commit -m "Mensaje descriptivo del cambio" # Add + commit en un paso (solo archivos ya trackeados) git commit -am "Mensaje" # Ver historial de commits git log git log --oneline git log --oneline --graph

4. Trabajar con Branches

# Ver ramas git branch # Crear rama git branch nueva-feature # Cambiar a rama git checkout nueva-feature # Crear y cambiar en un paso git checkout -b feature/login # Volver a main git checkout main # Fusionar rama en main git checkout main git merge feature/login # Eliminar rama git branch -d feature/login # Ver todas las ramas (locales y remotas) git branch -a

5. Repositorios Remotos

# Ver remotos configurados git remote -v # Añadir remoto git remote add origin https://github.com/usuario/repo.git # Subir rama al remoto git push -u origin main # Push normal (después de -u) git push # Descargar cambios del remoto git pull # Descargar sin fusionar git fetch # Ver cambios antes de hacer pull git fetch git diff origin/main

6. Resolver Conflictos

# Cuando hay conflicto al hacer merge o pull: # Git marca los conflictos en el archivo: <<<<<<< HEAD Tu código local ======= Código del remoto >>>>>>> origin/main # Pasos para resolver: # 1. Abrir archivo y editar manualmente # 2. Elegir qué código mantener # 3. Eliminar los marcadores (<<<, ===, >>>) # 4. Hacer commit del archivo resuelto git add archivo-conflicto.java git commit -m "Resolver conflicto en archivo"

7. Deshacer Cambios

# Descartar cambios en archivo (no commiteado) git checkout -- archivo.java # Quitar del staging area git reset HEAD archivo.java # Deshacer último commit (mantiene cambios) git reset --soft HEAD~1 # Deshacer último commit (descarta cambios) git reset --hard HEAD~1 # Crear commit que revierte otro git revert abc123

8. Flujo de Trabajo Completo

# 1. Actualizar main git checkout main git pull # 2. Crear rama para nueva funcionalidad git checkout -b feature/carrito-compra # 3. Desarrollar y hacer commits git add . git commit -m "Añadir modelo Carrito" git add . git commit -m "Implementar añadir producto al carrito" git add . git commit -m "Implementar calcular total" # 4. Subir rama al remoto git push -u origin feature/carrito-compra # 5. En GitHub: Crear Pull Request # 6. Después de aprobar, fusionar git checkout main git pull git merge feature/carrito-compra git push # 7. Eliminar rama git branch -d feature/carrito-compra git push origin --delete feature/carrito-compra

GitHub Básico

  • Fork: Copiar repositorio de otro usuario a tu cuenta
  • Pull Request: Solicitar fusión de tu rama en otra
  • Issues: Reportar bugs o solicitar features
  • Actions: CI/CD automatizado

Buenas prácticas: Commits pequeños y frecuentes, mensajes descriptivos en imperativo ("Añadir función", no "Añadida función"), no commitear contraseñas ni archivos sensibles.

Depuración (Debugging)

El debugging es el proceso de encontrar y corregir errores en el código. Los IDEs modernos ofrecen potentes herramientas de depuración.

Conceptos de Debugging

  • Breakpoint: Punto donde se pausa la ejecución
  • Step Over (F8): Ejecutar línea sin entrar en métodos
  • Step Into (F7): Entrar dentro del método
  • Step Out (Shift+F8): Salir del método actual
  • Resume (F9): Continuar hasta siguiente breakpoint
  • Watch: Observar valor de variables
  • Evaluate Expression: Evaluar expresiones en tiempo real

1. Ejemplo - Encontrar Error en Bucle

public class BusquedaArray { public static int buscar(int[] array, int objetivo) { // Poner breakpoint en esta línea for (int i = 0; i <= array.length; i++) { // ERROR: <= debería ser < if (array[i] == objetivo) { return i; } } return -1; } public static void main(String[] args) { int[] numeros = {5, 2, 8, 1, 9}; int resultado = buscar(numeros, 8); System.out.println("Índice: " + resultado); } } // Debug: // 1. Poner breakpoint en el for // 2. Ejecutar en modo debug // 3. Usar Step Over para ver cada iteración // 4. Observar valor de i en el panel de variables // 5. Cuando i=5 y array.length=5, veremos el error

2. Ejemplo - NullPointerException

public class GestorUsuarios { private List<Usuario> usuarios; public void agregarUsuario(Usuario u) { usuarios.add(u); // ERROR: usuarios nunca inicializado } public static void main(String[] args) { GestorUsuarios gestor = new GestorUsuarios(); gestor.agregarUsuario(new Usuario("Juan")); } } // Debug: // 1. Breakpoint en usuarios.add(u) // 2. Ver panel de variables: usuarios = null // 3. Solución: inicializar en constructor o declaración // private List<Usuario> usuarios = new ArrayList<>();

3. Ejemplo - Lógica Incorrecta

public class Calculadora { public static double calcularMedia(int[] notas) { int suma = 0; for (int nota : notas) { suma += nota; } return suma / notas.length; // ERROR: división entera } public static void main(String[] args) { int[] notas = {7, 8, 6, 9, 7}; double media = calcularMedia(notas); System.out.println("Media: " + media); // Imprime 7.0 en vez de 7.4 } } // Debug: // 1. Breakpoint en return // 2. Evaluate: suma = 37, notas.length = 5 // 3. Evaluate: suma / notas.length = 7 (división entera!) // 4. Solución: return (double) suma / notas.length;

4. Breakpoints Condicionales

public class ProcesadorPedidos { public void procesar(List<Pedido> pedidos) { for (Pedido p : pedidos) { // Click derecho en breakpoint > Condition // Condición: p.getTotal() > 1000 validar(p); calcularDescuento(p); enviar(p); } } } // El debugger solo para cuando el total > 1000 // Útil para listas grandes donde solo nos interesa un caso

Errores Comunes

  • NullPointerException: Usar objeto que es null
  • ArrayIndexOutOfBoundsException: Índice fuera de rango
  • StackOverflowError: Recursión infinita
  • Off-by-one: Error de uno en bucles

Testing con JUnit

Las pruebas automatizadas aseguran que el código funciona correctamente y facilitan el mantenimiento y refactorización.

Tipos de Tests

  • Tests Unitarios: Prueban métodos individuales
  • Tests de Integración: Prueban interacción entre componentes
  • Tests de Sistema: Prueban el sistema completo
  • Tests de Aceptación: Verifican requisitos del cliente

Metodologías Ágiles

Las metodologías ágiles priorizan la colaboración, la flexibilidad y la entrega continua de software funcionando.

Manifiesto Ágil (Valores)

  • Individuos e interacciones sobre procesos y herramientas
  • Software funcionando sobre documentación extensiva
  • Colaboración con el cliente sobre negociación de contratos
  • Respuesta al cambio sobre seguir un plan

Scrum

Framework ágil más popular para gestión de proyectos.

Roles en Scrum

  • Product Owner: Define requisitos, prioriza backlog
  • Scrum Master: Facilita el proceso, elimina obstáculos
  • Development Team: Desarrolla el producto (3-9 personas)

Eventos Scrum

  • Sprint: Iteración de 1-4 semanas
  • Sprint Planning: Planificar el trabajo del sprint
  • Daily Scrum: Reunión diaria de 15 min (stand-up)
  • Sprint Review: Demostración al final del sprint
  • Sprint Retrospective: Reflexionar y mejorar

Artefactos Scrum

  • Product Backlog: Lista priorizada de todo lo que necesita el producto
  • Sprint Backlog: Items seleccionados para el sprint actual
  • Incremento: Software funcionando al final del sprint

Kanban

Sistema visual de gestión del flujo de trabajo.

  • Tablero visual: Columnas To Do, In Progress, Done
  • WIP Limits: Límite de tareas en progreso
  • Flujo continuo: Sin sprints fijos
  • Mejora continua: Optimizar el flujo

Extreme Programming (XP)

Metodología enfocada en prácticas técnicas.

  • Pair Programming: Dos programadores en un ordenador
  • TDD: Test Driven Development
  • Integración Continua: Integrar código frecuentemente
  • Refactoring: Mejorar código sin cambiar funcionalidad
  • Simple Design: Diseño lo más simple posible

Herramientas de Gestión

  • Jira: Completo para Scrum y Kanban
  • Trello: Simple, visual, ideal para Kanban
  • GitHub Projects: Integrado con repositorios
  • Azure DevOps: Suite completa de Microsoft

/** * Gestiona las operaciones de una cuenta bancaria. * * @author Tu Nombre * @version 1.0 */ public class CuentaBancaria { private String titular; private double saldo; /** * Crea una cuenta bancaria con saldo inicial. * * @param titular nombre del titular de la cuenta * @param saldoInicial saldo inicial (debe ser >= 0) */ public CuentaBancaria(String titular, double saldoInicial) { this.titular = titular; this.saldo = saldoInicial; } /** * Deposita dinero en la cuenta. * * @param cantidad cantidad a depositar (debe ser > 0) * @throws IllegalArgumentException si cantidad <= 0 */ public void depositar(double cantidad) { if (cantidad <= 0) { throw new IllegalArgumentException("Cantidad debe ser positiva"); } saldo += cantidad; } /** * Retira dinero de la cuenta. * * @param cantidad cantidad a retirar * @return true si se pudo retirar, false si saldo insuficiente */ public boolean retirar(double cantidad) { if (cantidad > saldo) { return false; } saldo -= cantidad; return true; } /** * Obtiene el saldo actual. * * @return el saldo de la cuenta */ public double getSaldo() { return saldo; } }
